# Spec: Codex2Codex Continuation-First Recovery

## Objective
Build a continuation-first recovery protocol for `codex2codex` so transient worker interruptions from the same provider path are resumed on the same thread before restart/failure. This reduces false failures, stale review artifacts, and noisy fix waves while preserving strict artifact and scope validation.

## Users
- Primary: lead agent running `codex2codex` plans through `run_plan.py` / `run_wave.py`.
- Primary: worker runtime maintainer debugging meight worker execution.
- Secondary: review/fix/coding workers that must recover from interrupted turns without changing task scope.
- Secondary: future agents consuming `.codex/work/<slug>/tasks.md`, `review.md`, and artifacts.

## Problem
Recent `codex2codex` execution exposed several failure modes:
- Provider/network interruptions such as `404 no active credentials`, socket disconnects, stream failures, and app-server interruptions were treated too much like hard failures.
- Review workers could stall or fail after high token usage without durable runner summaries.
- PTY/session loss made runner outcome hard to reconstruct.
- Stale `review.md` FAIL and autogenerated fix waves remained after the underlying issue was fixed.
- Review infra failures could create code-fix workflow noise.
- Runner-side contract gates and validator gates can drift unless shared semantics are enforced.
- Worker tool/MCP failures can consume tokens without clean downgrade or durable diagnosis.

The desired behavior: transient interruption is normal. The runner should continue the child agent first, validate outputs strictly, and only escalate after bounded no-progress recovery.

## Success Criteria
- Transient provider/stream/tool interruptions default to same-thread continuation up to `3` times before restart/fresh-worker escalation.
- A continuation attempt that shows measurable progress does not immediately count as a terminal hard failure.
- No-progress attempts are bounded and eventually escalate to same-name restart, fresh worker, or terminal category.
- `wait exit 0` is not treated as success unless artifact, verdict, diff, and scope contracts pass.
- `wait exit 3` / unresolved `QUESTION:` cannot be hidden by validation PASS.
- Review worker infra failure does not create code fix waves.
- Stale review/fix artifacts can be identified and superseded.
- Runner state can be reconstructed after PTY/session loss from durable files.
- Runner and validator share exact-file vs directory-scope semantics.
- Full existing tests plus new recovery tests pass.

## Scope
### In
- Add continuation-first recovery behavior in `codex2codex/scripts/run_wave.py`.
- Persist worker attempt lifecycle in durable per-wave files.
- Distinguish transient interruption, no-progress continuation, contract failure, task blocker, and review infra unavailable.
- Add review infra degradation path that avoids fix-wave generation for infra-only review failures.
- Add stale artifact/fix-wave supersede metadata or sidecar state.
- Extract shared scope/contract helpers used by runner and validator.
- Add tests for recovery behavior, artifact supersession, stale fix-wave avoidance, and shared scope semantics.
- Keep existing CLI behavior compatible unless new flags are explicitly added.

### Out
- No default live provider smoke test. It can be opt-in only.
- No infinite retry.
- No silent lead implementation of worker-scoped code.
- No replacement of meight runtime.
- No broad rewrite of `run_plan.py` / `run_wave.py` beyond recovery, ledger, artifact, and contract boundaries.
- No auto-commit, branch workflow, or git history changes.
- No relaxation of artifact/review validation.

## Requirements
### Functional
- Classify transient interruptions as recoverable continuation events:
  - provider 404/no active credentials
  - socket disconnect/closed/timeout
  - app-server interruption
  - stream exception
  - provider timeout
  - retryable 429/5xx gateway errors
- On transient interruption, runner must first continue the same worker thread using `follow` or equivalent meight continuation.
- Default continuation budget must be `3` same-thread continuation attempts.
- Continuation attempt must include a compact recovery brief containing:
  - original worker name
  - failure category
  - observed reason
  - original task scope
  - required output artifact
  - instruction to continue original task, not restart from scratch
- Runner must track progress between attempts using available signals:
  - `updated_at` / `last_event_at`
  - token deltas
  - result tail changes
  - artifact existence/change
  - `files_changed`
  - status state transition
- If an attempt shows progress, runner may continue within budget instead of immediately escalating to restart.
- If repeated transient attempts show no progress, runner escalates in order:
  - same-thread continuation
  - same-name restart
  - fresh-worker restart
  - terminal category
- Terminal categories must remain explicit:
  - `INFRA_FAILED`
  - `CONTRACT_FAILED`
  - `TASK_BLOCKED`
  - `REVIEW_UNAVAILABLE`
- Review-only worker infra failure must produce `REVIEW_UNAVAILABLE`, not review `FAIL`.
- `REVIEW_UNAVAILABLE` must not create a fix wave.
- A review `FAIL` with valid artifact still creates/uses fix-wave flow unless `--no-fix-wave`.
- A superseded review artifact must make fix waves derived from it obsolete or ignored.
- Durable runner state must be written under the active work/spec directory, not only printed to PTY.
- Durable state must include enough data to reconstruct:
  - wave
  - worker
  - attempt number
  - worker name/thread id if available
  - category
  - action taken
  - progress signals
  - artifact path
  - final outcome
- Runner and validator must share scope semantics:
  - exact file scope allows only exact file path
  - explicit directory scope with trailing slash allows children
  - existing directory scope allows children
  - file-like scope must not allow child paths
- Artifact validation must reject:
  - missing artifact
  - empty artifact
  - blocked artifact
  - review artifact without `Verdict: PASS|FAIL`
  - implementation artifact without scoped `files_changed`
  - out-of-scope `PATCH_BODY`
  - no-op `PATCH_BODY`
- `PATCH_BODY` fallback remains runner-owned and scope validated.
- `ARTIFACT_BODY` salvage remains allowed only when contract can be validated.
- `run_plan.py` must forward any new recovery flags to `run_wave.py`.

### Non-Functional
- Backward compatible with existing `tasks.md` and generated manifests.
- Recovery state files must be human-readable JSON/JSONL or Markdown.
- Logs/artifacts must not include raw transcripts, secrets, credentials, or private provider payloads.
- New behavior must be covered by deterministic unit tests without requiring live provider calls.
- Existing `rtk python3 -m unittest discover codex2codex/scripts -p "test_*.py"` must pass.
- Dry-run must remain non-mutating except temporary dry-run dirs.

## Constraints
- Default same-thread continuation budget: `3`.
- Restart budgets must remain bounded; no unbounded loops.
- Existing recovery flags such as `--same-worker-restarts`, `--fresh-worker-restarts`, `--no-preflight`, and `--preflight-timeout` must keep working.
- Default preflight remains passive `meight doctor --json`.
- Optional live provider smoke, if added, must be opt-in and never required for normal test runs.
- Dirty worktree must be tolerated; runner must rely on scoped files and manifest, not global clean repo state.
- Lead final acceptance remains outside worker authority.

## Proposed Data Artifacts
- `.codex/work/<slug>/runs/<wave-slug>/<worker>/attempts.jsonl`
  - append-only attempt ledger
- `.codex/work/<slug>/runs/<wave-slug>/summary.json`
  - final durable outcome
- Optional artifact sidecar:
  - `<artifact>.meta.json`
  - fields: `run_id`, `wave`, `worker`, `attempt`, `source`, `verdict`, `supersedes`, `created_at`
- Optional obsolete marker:
  - `fix-wave-obsolete` state in `tasks.md` or `summary.json`

## Acceptance Checks
- Unit test: provider 404/no active credentials triggers same-thread continuation before restart.
- Unit test: socket disconnect/stream exception classified as transient continuation.
- Unit test: transient attempt with progress does not immediately terminal-fail.
- Unit test: repeated no-progress transient attempts escalate after budget.
- Unit test: review infra failure returns `REVIEW_UNAVAILABLE` and creates no fix wave.
- Unit test: valid review `FAIL` still creates fix wave.
- Unit test: superseded FAIL review invalidates or obsoletes derived fix wave.
- Unit test: PTY/session loss can be recovered from durable `summary.json` / `attempts.jsonl`.
- Unit test: runner and validator reject `src/target.py/child.txt` for `src/target.py` file scope.
- Unit test: runner and validator accept child path for explicit/existing directory scope.
- Unit test: `wait exit 3` remains non-success even if validation passes.
- Command: `rtk python3 -m unittest discover codex2codex/scripts -p "test_*.py"`
- Command: `rtk python3 codex2codex/scripts/run_plan.py .codex/work/20260619-codex2codex-worker-recovery/plan.md --dry-run`
- Manual check: force a fake transient worker status/result and verify attempts ledger records continue/restart/final outcome.

## Assumptions To Validate
- [ ] Same provider errors like 404/no active credentials can be transient in this environment.
  - Validate with fake meight tests first; optionally validate manually with real provider interruption.
- [ ] Same-thread continuation preserves enough context to finish worker tasks after stream interruption.
  - Validate by simulating interrupted result then successful follow.
- [ ] Progress signals are sufficient to avoid both premature failure and infinite waits.
  - Validate with tests for token delta, event delta, artifact delta, and no-progress loops.
- [ ] Review infra failures should not trigger code fix waves.
  - Validate with review-only manifest tests.
- [ ] Artifact sidecar/supersede metadata is enough to prevent stale review/fix-wave confusion.
  - Validate with stale FAIL -> new PASS scenario.

## Risks
- Transient recovery may wait too long on truly broken provider state.
  - Mitigation: bounded continuation budget and no-progress escalation.
- Progress heuristic may treat noisy token changes as useful progress.
  - Mitigation: combine token delta with event/result/artifact changes where possible.
- Same-thread continuation may repeat stale instructions.
  - Mitigation: compact recovery brief explicitly says continue original scope and produce final artifact.
- Artifact supersede may conflict with existing plain Markdown workflows.
  - Mitigation: use sidecar metadata first; keep Markdown readable.
- Review unavailable may be mistaken as PASS.
  - Mitigation: distinct `REVIEW_UNAVAILABLE` category and explicit final summary wording.
- Shared contract extraction may break imports/scripts.
  - Mitigation: keep helper small, stdlib-only, test both runner and validator.

## Open Questions
- Should `REVIEW_UNAVAILABLE` make `run_wave.py` exit nonzero by default, or exit with a distinct code while avoiding fix-wave creation?
- Should continuation budget be exposed as `--same-thread-continues` or folded into existing restart budget flags?
- Should artifact metadata be embedded frontmatter or sidecar JSON?
- Should stale fix waves be deleted, marked obsolete, or ignored by runner while preserving history?
- Should provider smoke be `--provider-smoke` on `run_wave.py`, `meight doctor`, or both?
