# Codex2Codex Context Firewall

## Problem Statement
How might `codex2codex` delegate heavy work to subagents so future lead agents see far less context, without accepting lower-quality implementation, review, or final decisions?

## Recommended Direction
Reframe `codex2codex` from a worker runner into a **context firewall execution protocol**.

The lead agent should not normally read raw worker transcripts, long result bodies, or repeated fix-loop details. Instead, every run produces a bounded `run-capsule.md/json`: scope, workers, changed files, verification, review verdicts, risks, unresolved questions, cleanup status, and evidence links. Raw outputs stay on disk behind artifact paths and hashes.

Quality must be protected by two layers: reviewer/gatekeeper workers inspect raw detail, while final validators verify receipts, artifacts, review PASS, provenance, and capsule consistency. For complex tasks, add a structured blackboard so workers interact meaningfully without pulling the lead into every exchange.

## Key Assumptions to Validate
- [ ] Lead can safely decide from capsule + evidence links in normal PASS cases. Test by replaying past `codex2codex` runs and measuring whether lead needs raw artifacts.
- [ ] Reviewer/gatekeeper workers can catch issues hidden by concise capsule summaries. Test with seeded defects and require review FAIL.
- [ ] Hard output budgets reduce lead tokens without hiding material risks. Test capsule size vs missed-defect rate.
- [ ] Blackboard improves complex-task quality enough to justify latency. Test only on multi-wave/high-risk plans first.
- [ ] Adaptive depth avoids making simple tasks too slow. Test routing heuristics on trivial/medium/complex examples.

## MVP Scope
- Add canonical `run-capsule.json` and `run-capsule.md`.
- Capsule includes fixed sections only: intent, waves, workers, artifacts, changed files, verification, review verdict, risks, blockers, evidence ledger, cleanup.
- Add capsule validator: no missing artifacts, no unverified PASS, no raw transcript leakage, all claims link to evidence.
- Update `run_plan.py` / `run_wave.py` to return capsule path as primary lead-facing output.
- Keep `validate_execution_complete.py` as final acceptance gate, but extend it to check capsule consistency.
- Add role prompt contract: workers write concise artifacts; reviewer/gatekeeper inspects raw detail; lead reads capsule first.
- Add context budget limits for worker summaries and review summaries.
- Add blackboard only for opt-in complex mode, not default MVP.

## Not Doing (and Why)
- **Streaming worker logs to lead** -- defeats the main goal.
- **Replacing review with final validator only** -- validators prove structure/completion, not semantic quality.
- **Making every task a council** -- too slow; use adaptive depth.
- **Letting lead synthesize missing worker/review work** -- reintroduces context overload and quality drift.
- **Dashboard/UI first** -- protocol and artifacts matter more than presentation.
- **Raw transcript summarization as the main product** -- summaries without evidence become hallucination surfaces.
- **Optimizing speed first** -- user explicitly accepts slower execution if quality and context isolation improve.

## Open Questions
- What exact capsule size budget is acceptable? e.g. 2k, 4k, 8k tokens.
- Which task-risk heuristic triggers blackboard/council mode?
- Should capsule be generated by runner code only, or by a gatekeeper worker plus validator?
- What is the minimum evidence schema that catches real quality failures without creating bureaucracy?
- Should failing runs expose more detail automatically, or still require explicit lead pull?
